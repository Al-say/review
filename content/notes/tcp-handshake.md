# TCP 三次握手详解

TCP（Transmission Control Protocol）是互联网的核心协议之一，其连接建立过程通过"三次握手"来确保可靠的通信。

## 握手过程

### 1. 第一次握手 (SYN)
客户端发送 SYN 包到服务器，进入 SYN_SENT 状态。

```
客户端 → 服务器: SYN=1, seq=x
```

### 2. 第二次握手 (SYN+ACK)
服务器收到 SYN 包后，回复 SYN+ACK 包，进入 SYN_RCVD 状态。

```
服务器 → 客户端: SYN=1, ACK=1, seq=y, ack=x+1
```

### 3. 第三次握手 (ACK)
客户端收到 SYN+ACK 包后，发送 ACK 包，进入 ESTABLISHED 状态。

```
客户端 → 服务器: ACK=1, seq=x+1, ack=y+1
```

服务器收到 ACK 包后也进入 ESTABLISHED 状态。

## 为什么需要三次握手？

三次握手的主要目的是**防止已失效的连接请求报文突然又传送到了服务器**，从而产生错误。

### 场景分析
假设客户端发送了一个 SYN 包，但由于网络延迟，这个包在网络中滞留了很久。客户端超时后重新发送了 SYN 包并建立了连接。

如果只使用两次握手，当那个延迟的 SYN 包最终到达服务器时，服务器会认为客户端又发起了一次新的连接请求，从而建立一个"幽灵连接"。

## 状态变迁

| 阶段 | 客户端状态 | 服务器状态 |
|------|-----------|-----------|
| 初始 | CLOSED | LISTEN |
| 1次握手后 | SYN_SENT | LISTEN |
| 2次握手后 | SYN_SENT | SYN_RCVD |
| 3次握手后 | ESTABLISHED | ESTABLISHED |

## 常见问题

### 1. 为什么不是两次握手？
防止历史连接的 SYN 包干扰当前连接。

### 2. 为什么不是四次握手？
三次握手已经能够满足可靠连接建立的需求。

### 3. SYN 洪泛攻击
攻击者发送大量 SYN 包但不回应 ACK，导致服务器维护大量半连接，消耗资源。

## 防御措施

1. **SYN Cookie**: 服务器不维护半连接队列，而是通过加密算法验证客户端的合法性。
2. **防火墙过滤**: 限制 SYN 包的速率。
3. **增加队列长度**: 提高服务器处理能力。

## 总结

TCP 三次握手是网络通信的基础，通过这种机制确保了连接的可靠建立。理解握手过程有助于我们更好地调试网络问题和优化系统性能。
